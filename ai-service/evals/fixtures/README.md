# Eval Fixture Pool

20 synthetic athletes used by Phase 2+ eval suites (safety_live, behavior_live)
to run scenarios against realistic athlete contexts without touching production
user data.

## Status (Phase 0, 2026-04-22)

- `pool.py` — **shipped.** Declarative fixture data, zero side effects.
  Import-time invariants assert coverage (20 fixtures, every axis-value >=3
  times, unique slugs/UUIDs, UUID prefix).
- `seed.py` — **deferred to Phase 2.** Will insert fixtures into
  `public.users` (via Supabase admin API to create `auth.users` first),
  then seed satellite tables per scenario needs. Exact satellite set
  depends on which scenarios Phase 2 adds.
- `__init__.py` — module re-exports.

## Why not seed now

Writing `seed.py` in Phase 0 would mean guessing which satellite tables
(check-ins, `player_schedule_preferences`, training_blocks, etc.) each
future eval scenario needs. Phase 2 adds the scenarios; we'll write
`seed.py` with verified requirements then, not speculated.

Until `seed.py` ships, the `users.is_eval_fixture` column exists (migration
092) but has zero rows. The RESTRICTIVE RLS policy on `public.users`
(`users_restrict_fixtures`) is live but has nothing to restrict — a no-op
until fixtures seed.

## Coverage matrix

| Axis | Values | Fixture count |
|---|---|---|
| Age band | U13 / U15 / U17 / U19 | 4 / 6 / 5 / 5 |
| PHV stage | pre_phv / mid_phv / post_phv / unknown | 3 / 7 / 9 / 1 |
| Sport | football / padel / basketball / tennis / athletics | 7 / 3 / 3 / 4 / 3 |
| Readiness | green / amber / red | 7 / 9 / 4 |

U15 mid-PHV gets the heaviest sampling because it's the most safety-critical
combination — highest PHV-stage injury risk, active loading multiplier (0.70),
active contraindications list. Every safety-specific eval should prefer these
fixtures.

`unknown` PHV appears exactly once (`u15_unknown_phv_football_amber`) to
exercise the "default to mid_phv protections" fallback per the skill's Layer 3
missing-data rules.

## Stable UUIDs

All fixture IDs use the namespaced prefix `00000000-1001-<ageband>-<seq>`
so they are visually distinguishable from production UUIDs generated by
`gen_random_uuid()`. Never collides in practice.

## Usage (Phase 2+)

```python
from evals.fixtures import EVAL_ATHLETES, get_by_slug

# Iterate for bulk scenarios
for athlete in EVAL_ATHLETES:
    run_scenario(athlete)

# Reference a specific fixture from a scenario spec
fixture = get_by_slug("u15_mid_phv_football_amber")
```

## Invariants (enforced at import)

- Exactly 20 fixtures
- All slugs unique
- All UUIDs unique and use FIXTURE_PREFIX
- Every age band appears >=3 times
- Every sport appears >=3 times
- Every readiness band appears >=3 times
- Every PHV stage appears >=1 time

If you edit `pool.py`, these asserts run on every import. A broken matrix
fails CI before evals run.

## Schema translation notes (for Phase 2 seed.py)

- `sport: "football"` → maps to `users.sport = "soccer"` (current CHECK
  constraint from migration 001 allows `soccer|basketball|tennis|padel`).
  If `athletics` is needed in `users.sport` CHECK, extend via new migration.
- `readiness_band` is derived from `readiness_score` (>=70 green, 40-69
  amber, <40 red); seed.py writes the score to `checkins`, band is computed
  downstream.
- `phv_stage: "unknown"` → `users.phv_stage = NULL`; readers default to
  `mid_phv` protections per Layer 3 rule.
- `ccrs` goes to the athlete snapshot row, not `users` directly.
- `academic_stress_score` goes to `checkins.academic_stress` or the
  dual-load computation input.

## Non-production guarantee

Every seeded row carries `is_eval_fixture = TRUE`. The RESTRICTIVE RLS
policy `users_restrict_fixtures` (migration 092) blocks anon/authenticated
reads of fixture rows. Service-role bypasses RLS (by design, for admin and
eval code). Product and analytics code must read `public.users_real` view,
which filters fixtures out at the view level.
